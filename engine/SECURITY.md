# Jellyclaw Engine — Security Model

**Status:** Living document. Last reviewed: 2026-04-14.
**Scope:** The jellyclaw engine (OpenCode fork) as run locally by Genie and as an embedded HTTP server for the Genie desktop shell.
**Audience:** Contributors, security researchers, operators deploying jellyclaw in shared or semi-trusted environments.

Jellyclaw is a fork of OpenCode, hardened for the Genie 2.0 stack. This document describes the threat model, the defense-in-depth stack, specific mitigations for known upstream issues, and our disclosure and update policies.

---

## 1. Threat Model

Jellyclaw runs in two distinct modes, each with a different threat surface.

### 1.1 Mode A — Local subprocess (default)

The Genie dispatcher spawns jellyclaw as a child process, pipes prompts over stdin, and reads tool calls and model responses over stdout. No network server is exposed. The threat surface is:

- **Prompt injection** from untrusted content pulled into context (web pages, emails, files). An attacker-controlled document can attempt to instruct the model to exfiltrate secrets, run destructive shell commands, or open network sockets.
- **Tool-exfiltration**: the model has `bash`, `read`, `write`, `fetch`, MCP browser tools. Any of these, if abused, leaks data to the attacker's host.
- **Supply chain**: transitive dependencies (npm packages, MCP servers) could contain malicious code that runs with the user's privileges.

Assumptions: the local user is trusted. The OS account jellyclaw runs under is trusted. We do not defend against a compromised host OS.

### 1.2 Mode B — HTTP server (opt-in, used by desktop shell)

Jellyclaw can expose an HTTP server for the Genie desktop app and external integrations. The threat surface expands dramatically:

- **CVE-2026-22812 class attacks** — any webpage the user visits in any browser can probe `http://127.0.0.1:<port>` and, absent auth, issue requests to `/session/*/shell`, `/pty`, `/file/content`. This is remote code execution by browser cross-origin request. CVSS 8.8. See §3.1.
- **Cross-protocol attacks** (DNS rebinding, WebSocket-via-HTTP smuggling) against the local bind.
- **Credential theft** if auth tokens are logged, broadcast on stderr, or embedded in shell history.

Assumptions in Mode B: the user's browser is hostile-by-proxy (any ad, any CDN, any tracker can make same-origin requests to `127.0.0.1`). The only trusted caller is the Genie desktop shell, authenticated by a per-session bearer token.

### 1.3 Mode C — Subagent task execution

When the model calls the `task` tool, it spawns a subagent. The subagent has its own tool loop. Upstream OpenCode issue #5894 confirms plugin hooks do not fire for tools the subagent calls. This means any security policy enforced via hooks (audit log, approval gate, secret scrub) is bypassed by simply asking the model to delegate the dangerous tool call to a subagent. We treat this as a **critical gap** and patch it — see §3.2.

## 2. Defense-in-Depth Stack

Layers below are ordered from network-edge inward. A bypass of one layer should not compromise the others.

### 2.1 Binding

**Policy:** Jellyclaw binds only to `127.0.0.1` (and `::1`). Binding to `0.0.0.0`, a public interface, or a LAN address is refused with an error at startup.

Override: an explicit `--unsafe-bind-all` flag is required to relax this, and when used jellyclaw prints a startup banner warning and writes an audit-log entry. See `patches/002-bind-localhost-only.patch`.

### 2.2 Authentication

**Policy:** Every HTTP request must carry `Authorization: Bearer <token>`. The token is a 32-byte random hex string, generated by the parent process (Genie dispatcher) at server startup and passed to jellyclaw via the `OPENCODE_SERVER_PASSWORD` env var. If not set, jellyclaw generates one and writes it to stderr exactly once.

No token, no origin check, no localhost shortcut: authentication is mandatory even from `127.0.0.1`. This defeats browser-origin probing (the browser has no way to learn the token).

Rotation: tokens are session-scoped. Restarting jellyclaw rotates.

### 2.3 CORS

**Policy:** The default `Access-Control-Allow-Origin` allowlist is **empty**. No origin may make a CORS-permitted request. The desktop shell uses a same-process IPC path or an explicit origin in `OPENCODE_ALLOWED_ORIGINS` (comma-separated, exact match, no wildcards).

Preflight `OPTIONS` requests return `403` unless the origin matches the allowlist. No `Access-Control-Allow-Credentials: true` under any configuration — the bearer token is enough.

### 2.4 Secret isolation

**Policy:** Secrets required by tools (Anthropic key, GitHub PAT, Telegram bot token, MCP server credentials) are held by the parent dispatcher, not injected wholesale into the jellyclaw subprocess env.

- The jellyclaw process env is filtered at spawn time (`PATH`, `HOME`, `USER`, `XDG_*`, `TMPDIR`, `LANG` only).
- Tool calls that need a secret request it from the dispatcher via a narrowly-scoped IPC RPC (`get_secret(name, scope)`), which mediates and logs.
- No tool called by the model directly reads `process.env.ANTHROPIC_API_KEY`. The dispatcher attaches auth headers on outbound HTTP calls so the key never crosses into model-visible tool memory.
- Telegram specifically (Genie feature): the LLM never sees `TELEGRAM_BOT_TOKEN`. Telegram delivery is a post-hook emission — the model emits a structured `emit:telegram` tool call, the hook on the dispatcher side fulfills it. See §3.5.

### 2.5 Sandboxing (optional but recommended)

**Policy:** The `bash` tool can be optionally wrapped by an OS-level sandbox.

- **macOS:** `sandbox-exec -f <profile>` with a profile that denies network (except explicit allowlist), denies file writes outside `$PWD` and `$TMPDIR`, allows read of system libraries.
- **Linux:** `bwrap` with `--unshare-net`, `--ro-bind` for `/usr`, `--bind` for cwd and tmpdir, `--die-with-parent`.
- **Windows:** no reliable equivalent shipped yet — Genie on Windows documents this gap.

Sandboxing is off by default (breaks many legitimate workflows like `git push`, `npm install`) but can be turned on globally via `jellyclaw.sandbox.bash = "strict"` in the engine config.

### 2.6 Rate limiting

**Policy:** Per-domain token-bucket rate limits on browser and fetch tools. Default: 30 requests/minute per origin, 4 parallel navigations. Exceeding triggers a retry-after response to the model, and three consecutive bucket exhaustions short-circuit the agent with an error. This prevents a runaway or prompt-injected loop from DoSing a third-party site.

### 2.7 Credential scrubbing on tool results

**Policy:** Before any tool result is handed back to the model, the bytes are passed through a credential-pattern redactor. Patterns: Anthropic `sk-ant-*`, OpenAI `sk-*`, OpenRouter `sk-or-*`, GitHub `ghp_*` / `gho_*` / `ghs_*`, Stripe `sk_live_*` / `rk_live_*`, AWS `AKIA*`, Google `AIza*`, JWT-shaped `eyJ[A-Za-z0-9_-]+\.eyJ[...]\.[...]`, `Bearer <token>`, and `.env`-style `KEY=value` lines for sensitive key names (`*_TOKEN`, `*_KEY`, `*_SECRET`, `PASSWORD`, `DATABASE_URL`).

Matches are replaced with `[REDACTED]` and an audit counter is incremented. The pattern set is derived from `/Users/gtrush/Downloads/genie-2.0/engines/hermes-agent/tools/mcp_tool.py` `_CREDENTIAL_PATTERN` and expanded. See `patches/003-secret-scrub-tool-results.patch`.

This is best-effort, not a substitute for §2.4 secret isolation. It catches secrets that leak through `cat .env`, accidental logging, or misconfigured tools.

### 2.8 Plugin hooks fire on subagent tool calls

**Policy:** `tool.execute.before` and `tool.execute.after` hooks fire for every tool invocation, including those made by subagents spawned via the `task` tool. The patched subagent code path threads the parent plugin context into the child loop. See `patches/001-subagent-hook-fire.patch`.

This closes the OpenCode #5894 bypass and means a single hook-based policy (e.g., "reject `bash` with `curl` to non-allowlisted host") applies uniformly.

## 3. Specific Mitigations

### 3.1 CVE-2026-22812 (OpenCode unauthenticated RCE)

**Upstream fix:** OpenCode 1.0.216 added auth; 1.1.10 disabled the server by default.

**Jellyclaw mitigations, stacked:**
- Server off by default (`--serve` required to enable).
- `127.0.0.1`-only bind enforced in code, not just config (patch 002).
- Mandatory bearer auth, empty CORS allowlist (patch 002 + built-in).
- All mutating endpoints (`/session/*/shell`, `/pty`, `/file/content`, `/file/write`) require an additional "confirm" header `X-Jellyclaw-Intent: execute` that matches the per-request intent hash. Browser same-origin requests cannot forge the intent hash.

Even if one of these fails, the others prevent RCE.

### 3.2 OpenCode #5894 (subagent hook bypass)

Patched in `patches/001-subagent-hook-fire.patch`. We intend to upstream this.

### 3.3 OpenRouter caching breakage (#1245, #17910)

Not a security issue but a cost-attribution issue. Addressed in `PROVIDER-STRATEGY.md`: route Anthropic models through the direct Anthropic API so `cache_control` works correctly. OpenRouter is a non-Anthropic fallback only.

### 3.4 Playwright MCP regression

Pinned to `@playwright/mcp@0.0.41` in the default `mcp.json` template. See `patches/004-playwright-mcp-pin.md`.

### 3.5 Telegram token leakage (Genie legacy)

The legacy `config/genie-system.md` instructed the LLM to `curl https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/...` from the `bash` tool. That forced the token into the jellyclaw subprocess env and made it visible to any tool the model drove.

**Migration:** the LLM now emits a structured `emit_notification` tool call with channel=telegram. The dispatcher holds the token and fulfills the emission in the parent process. The jellyclaw subprocess never sees `TELEGRAM_BOT_TOKEN`. Implemented as a plugin hook; the system prompt has been rewritten to document only the high-level API.

## 4. Incident Response

If a CVE is discovered in jellyclaw or in an upstream OpenCode path we ship unmodified:

1. **Triage (within 24h):** confirm reproducibility, assess CVSS, classify scope (local vs. server-mode vs. supply-chain).
2. **Containment:** if actively exploited, release a silent pre-announcement update to the stable channel that disables the affected code path (e.g., `--serve` refuses to start, or the vulnerable tool is gated behind a new approval prompt).
3. **Patch:** backport upstream fix if available; otherwise write and test our own.
4. **Disclosure:** coordinate with upstream OpenCode security team. Honor 90-day disclosure windows unless in-the-wild exploitation warrants earlier publication.
5. **User notification:** changelog entry, banner in the desktop shell on next launch, email to operators who opted into security announcements.
6. **Postmortem:** published within 14 days for any issue CVSS >= 7.0.

Security contact: security@gtru.xyz (PGP key in repo root).

## 5. Telemetry & Privacy

Jellyclaw collects, by default:

- Anonymized model name and token counts per session (for cost attribution).
- Tool name frequency (not arguments, not results).
- Crash stack traces with source paths but without file contents or env values.

Jellyclaw never collects: prompts, responses, tool arguments, tool results, file contents, user identifiers, IP addresses, MAC addresses. The dispatcher does not phone home on startup. Telemetry is batched and emitted only when the user has `jellyclaw.telemetry = "on"` in config; default is `"off"`.

Opt-out: set `jellyclaw.telemetry = "off"` or launch with `--no-telemetry`. No environment variable override can force telemetry on.

## 6. Update Channel & CVE Policy

- Release channels: `stable`, `beta`, `nightly`.
- `stable` is the default; security backports land within 72h of upstream or our own disclosure.
- CVEs are published to `https://jellyclaw.dev/security/` with CVSS, affected versions, mitigations, and upgrade path.
- We participate in the GitHub Security Advisory process and request CVE IDs via the GitHub CNA for jellyclaw-specific issues.
- Downstream forks of jellyclaw are asked to subscribe to the advisory stream.

---

## 7. Non-Goals

- Jellyclaw does not attempt to sandbox the model against the operator. An operator who gives the model shell access accepts that prompt injection can drive shell commands. Our defenses raise cost and blast radius, not eliminate intent.
- Jellyclaw does not encrypt at rest. Session state lives in `$XDG_STATE_HOME/jellyclaw/` and inherits filesystem permissions. Full-disk encryption is the user's responsibility.
- Jellyclaw is not certified for regulated workloads (HIPAA, PCI). Use is at the operator's risk.
